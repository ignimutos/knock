# 测试架构与硬门禁设计（一次性改造）

日期：2026-04-14  
范围：全仓库测试层（含 `src/**`、`web/**`、`infrastructure/**` 相关测试）

## 1. 背景与目标

当前测试模块规模大、组织分散、setup 方式不统一，导致以下结果：

- 关键回归路径覆盖可见性弱，问题命中率不稳定
- 测试代码维护成本高，新增功能时改动面扩散
- AI 改测试时缺少统一执行规范，结果质量波动

本次改造采用一次性方案，优先级为：

1. 关键回归场景命中率提升（A）
2. 测试执行效率提升（B）
3. 维护成本下降（C）

## 2. 设计原则

1. 单一事实源：风险矩阵作为“测什么”的统一入口
2. 分层清晰：`unit / contract / flow` 三层职责固定
3. 共享夹具：测试 runtime、fixture、断言工具统一收敛
4. 门禁前置：改测试即触发检查，缺项立即阻断
5. 最小验证优先：默认 scoped 验证，命中共享高风险边界再扩大

## 3. 目标测试架构

### 3.1 三层模型

- `unit`
  - 目标：验证纯逻辑与小模块行为
  - 特征：零 I/O、快速执行、最小依赖
- `contract`
  - 目标：验证模块边界契约
  - 覆盖：配置 shape、错误文案契约、adapter I/O 契约、OTel 字段归属
- `flow`
  - 目标：验证跨模块关键路径与回归风险
  - 覆盖：source -> pipeline -> delivery -> state 的端到端行为片段

### 3.2 组织策略

- 按业务域聚合测试，在每个域内保持三层分组
- 共享测试支撑集中到统一测试工具域
- 业务测试文件只通过统一入口构造 runtime/fixture/assertion

### 3.3 命名与可追踪

- 测试命名格式：`[层级] [场景] [期望结果]`
- 每个 `flow` 用例绑定风险矩阵 ID（例如 `R07-delivery-retry-exhausted`）
- 断言优先验证可行动信号：错误类别、关键字段、状态变化、边界结果

## 4. 共享测试组件设计

### 4.1 TestRuntimeHarness

职责：统一生命周期、隔离资源、减少重复 setup/cleanup。

能力：

- runtime 目录准备与清理
- 临时资源隔离（文件、数据库、上下文）
- 统一 teardown 行为

### 4.2 FixtureFactory

职责：提供可组合、可覆盖的最小业务构造器。

覆盖对象：

- source/config/entry/delivery/db-state
- 默认值可直接运行，支持显式 override

### 4.3 AssertionKit

职责：提供统一断言语义，提升失败可读性与稳定性。

覆盖断言：

- 错误分类与关键错误字段
- OTel 结构字段位置与级别一致性
- 状态迁移与输出副作用

### 4.4 ScenarioRunner

职责：驱动 `flow` 用例的标准执行管道。

标准流程：

1. 注入前置状态
2. 执行目标路径
3. 采集观测点
4. 进行分层断言

## 5. 风险矩阵（Top 20）

### 5.1 固定字段

每条风险条目固定以下字段：

- `id`：`R01`..`R20`
- `domain`：所属业务域
- `trigger`：触发条件
- `expected_guardrail`：期望保护行为
- `required_layer`：必须覆盖层级（如 `flow+contract`）
- `owner_tests`：归属测试文件列表

### 5.2 覆盖要求

- 所有 Top 20 风险都必须存在归属测试
- 每个风险至少命中 `required_layer`
- 风险条目与测试映射可双向追踪

### 5.3 首批风险域分配

- source 抓取与解析
- pipeline 过滤/去重/渲染
- delivery 发送/重试/最终失败
- config 解析与契约边界
- sqlite 状态一致性与恢复路径
- web playground 评估链路

## 6. 硬门禁设计（AI 自动参考）

### 6.1 触发范围

命中以下任一改动即触发门禁：

- 测试文件（`**/*_test.ts`、`**/test_*.ts`）
- 测试工具文件（如 runtime harness、fixture、assertion、scenario runner）
- 被测试直接依赖的测试工具文件

### 6.2 门禁检查顺序

1. 风险映射检查：改动是否关联风险 ID
2. 共享入口检查：是否使用统一 harness/factory/assertion
3. scoped 验证检查：是否运行最小相关 `test/check/lint:check/fmt:check`
4. 高风险边界检查：若命中共享高风险边界，是否补跑全量 `deno task test`

### 6.3 失败输出结构

门禁失败统一输出：

- `gate`
- `failed_checks[]`
- `actionable_fix[]`
- `related_paths[]`

每个失败项都提供直接修复动作。

## 7. Skill 与规则落地

### 7.1 仓库规则

新增项目级测试规范文档，定义：

- 三层模型与命名规则
- 风险矩阵维护规则
- 门禁触发范围与阻断条件
- 验证命令策略与报告格式

### 7.2 Skill 设计

新增可调用 skill（建议名：`test-architecture-guard`），支持两种触发：

- 默认触发：AI 检测到测试相关改动时自动应用
- 手动触发：`/test-architecture-guard`

Skill 核心流程：

1. 识别改动路径并判定是否触发门禁
2. 引导补齐风险映射与共享入口
3. 执行 scoped 验证并收集结果
4. 产出可审计门禁报告

## 8. 一次性迁移执行方案

1. 建立共享测试组件骨架（Harness/Factory/Assertion/Runner）
2. 建立 Top 20 风险矩阵文件与映射机制
3. 全仓测试迁移到三层模型并接入共享入口
4. 补齐风险缺口用例
5. 接入硬门禁 skill 与仓库规则
6. 运行分层验证与必要全量验证
7. 输出迁移报告（覆盖、耗时、维护性变化）

## 9. 验收标准

### 9.1 A：关键回归命中率

- Top 20 风险全部建立映射并有可执行用例
- 每个风险具备明确 `expected_guardrail`
- `flow` 失败可直接定位到风险 ID

### 9.2 B：执行效率

- 默认开发路径使用 scoped 验证
- 全量验证仅在命中共享高风险边界时触发
- 迁移报告记录基线与改造后对比

### 9.3 C：维护成本

- 新增测试优先复用共享夹具
- 测试变更集中在领域测试文件与共享工具
- 重复 setup 模板显著收敛

## 10. 风险与控制

- 风险：一次性改动面大，迁移期回归压力上升  
  控制：按业务域分批提交，持续保持门禁可运行
- 风险：团队短期不适应新分层规范  
  控制：在 skill 输出中内置改造提示与示例路径
- 风险：风险矩阵维护滞后  
  控制：门禁将“新增高风险改动无矩阵条目”设为阻断项

## 11. 非目标

- 本次改造不调整生产业务行为与对外契约
- 本次改造不引入新的配置 shape
- 本次改造不改变既有中文错误文案契约

## 12. 实施后交付物

- 三层测试架构与迁移后的测试文件
- 共享测试组件（Harness/Factory/Assertion/Runner）
- Top 20 风险矩阵与测试映射
- 项目级测试规范文档
- `test-architecture-guard` skill
- 迁移验证报告（通过项、未运行项、剩余风险）
